# 확장 지시서 v7: KB 전략 확정 (제품 목표 기반) + 미결 답변 반영

> 이 문서는 v6(표 추출기 → 이벤트 하이브리드)와 **함께** 쓰인다. v6의 시간/날짜 처리에 아래 KB 전략을 적용하고, 그동안 미결이던 두 질문의 답을 확정한다.

---

## 0. 확정 사항

1. **제품 목표 확정:** 이 제품은 **학생이 현재·다가올 학기 시간표를 짜는 도구**다. 2005~2026 과거 학기를 정확한 날짜로 복원하는 역사 아카이브가 **아니다.** → 이 목표가 아래 KB 전략의 전제다.
2. **SYL-028 수업시간:** **확정(Y)으로 처리**한다. gold에 포함하고 method_report 재집계에 반영하라.
3. **§8-4 KB 범위:** 아래 §1~§6 전략대로. 요약하면 "교시표는 넓게, 학사일정은 최소한, 문서 내 날짜 최우선".

---

## 1. KB 전략 대원칙 (2인 인력 기준으로 최적화)

인력이 2명이므로 KB 채우기(수작업 리서치)를 롱테일에 태우면 안 된다. 두 KB는 비용 구조가 다르니 다르게 다룬다.

- **교시표 KB — 넓게.** 키가 (학교, 캠퍼스)뿐이고 연도 무관, 싸고 레버리지가 크다. 코퍼스 대다수를 덮는 상위 학교만 채우면 된다.
- **학사일정 KB — 최소한.** 키가 (학교 × 학년도 × 학기)라 조합이 폭발하고, 과거 학사일정은 공개 안 된 경우가 많다. **과거 학기는 채우지 않는다.** 현재·다가올 학기 + 상위 학교만.
- **문서 내 날짜 최우선.** 많은 실라버스(특히 연세 계열)는 주차별 표에 실제 날짜가 이미 있다(SYL-022 "2015-03-02~03-08", SYL-024 "2014-09-01", SYL-028 "3/4-3/8"). 이 경우 학사일정 KB가 아예 불필요하다. v6의 표 추출기가 이 날짜 컬럼을 뽑으면 "Week N → 실제 날짜"가 공짜로 된다.
- **범위 밖은 needs_review.** 교시표 없는 학교, 학사일정 없는(과거) 학기는 raw(교시/주차)로 남기고 needs_review. 실패가 아니라 올바른 안전 동작(precision > recall).

---

## 2. 해석 우선순위 (resolver 로직 — 반드시 이 순서)

**날짜 (주차/일차 → 실제 날짜):**
1. **문서 내 명시 날짜** (표 기간 컬럼 등) → 그대로 사용 (`resolved_by: in_document`). ← 최우선, 과거 문서도 공짜로 해결
2. 없고 **(현재·다가올 학기 + 상위 학교)면** → 학사일정 KB로 변환 (`resolved_by: academic_calendar_kb`)
3. 그 외 → `resolved_date=null`, `date_kind=relative` 유지, `needs_review=true`. **임의 날짜 생성 절대 금지**(SYL-032 환각 재발 방지)

**시각 (교시 → 실제 시각):**
1. **문서 내 시각 병기** ("2교시(10:00-10:50)") → 그대로 사용 (`resolved_by: in_document`)
2. 없고 **교시표 KB에 (학교,캠퍼스) 있으면** → 변환 (`resolved_by: period_timetable_kb`)
3. 그 외 → `resolved_time=null`, `needs_review=true`

## 3. KB 파일 구조 (config)

```yaml
# config/period_timetables.yaml  — 상위 학교로 넓게. 연도 무관.
timetables:
  yonsei_seoul:   { periods: { "1":["09:00","09:50"], "2":["10:00","10:50"], ... } }
  yonsei_wonju:   { periods: { ... } }        # 원주는 서울과 다름 (SYL-024)
  korea:          { periods: { ... } }
  hanyang:        { periods: { ... } }
  # 롱테일 학교는 넣지 않음 → needs_review
```
```yaml
# config/academic_calendars.yaml  — 현재·다가올 학기만. 과거 채우지 마라.
calendars:
  yonsei__2026_2:  { term_start: "2026-09-01", term_end: "2026-12-18",
                     holidays: [...] }
  korea__2026_2:   { term_start: "2026-09-01", holidays: [...] }
  # 과거 학기(2005~2024)는 의도적으로 비움. 문서 내 날짜 또는 needs_review로 처리.
```
- `academic_calendars.yaml`은 **학기마다 갱신**하는 소량 파일로 취급(현재+다음 학기만 유지). 과거로 소급 채우지 마라.

## 4. 빌드 순서 (2인 인력 실제 작업)

1. **Claude Code (사람 손 0):** 코퍼스 1,026건의 **학교·캠퍼스 분포**를 뽑아라. "상위 N개 학교 = 커버리지 %" 곡선을 보고하라. 이 곡선이 교시표를 어디까지 채울지 자동으로 정한다.
2. **사람 2인 (1~2인일):** 위 곡선에서 **코퍼스 ~80~85%를 덮는 상위 학교·캠퍼스 교시표만** 채운다(대략 8~12개 항목 예상). 여기서 멈춘다.
3. **학사일정 (필요 시 소량):** **현재·다가올 학기 + 상위 학교만**. 과거는 손대지 않는다. 문서 내 날짜가 대부분을 해결하므로 이건 보조.
4. 나머지는 needs_review. 롱테일에 시간 쓰지 마라.

---

## 5. 중단 규칙 & KB 커버리지 지표

- **중단 규칙:** 상위 학교 교시표를 채운 뒤, **현재·다가올 학기 문서의 needs_review 비율**을 재라. 감당 가능하면 거기서 멈춘다. precision > recall이므로 needs_review가 남는 건 정상.
- **커버리지 지표(보고):** 해결 경로별 비율을 분리 집계 — 문서 내 날짜 %, 교시표 KB %, 학사일정 KB %, needs_review %.
- **과거/OCW 문서의 needs_review는 정상**으로 집계하라(제품 목표 밖). 이걸 실패로 세지 마라. 지표는 "현재·다가올 학기 문서" 기준으로 봐야 의미가 있다.

---

## 6. 평가·회귀 테스트에 반영

- 문서 내 날짜가 KB보다 **우선**되는지 (표에 날짜 있는 문서는 KB 없이도 resolved).
- 범위 밖 학교 교시 → needs_review (임의 시각 생성 0).
- 과거 학기 주차-only → needs_review (임의 날짜 생성 0, SYL-032 유형).
- resolver 정확도는 모델/분류 정확도와 **분리 측정**: "raw→resolved 변환이 맞았나"만 따로.

---

## 7. v6와의 연결

- v6 표 추출기의 **날짜 컬럼 추출이 학사일정 KB의 필요를 크게 줄인다.** 그러니 §2 우선순위에서 in_document를 최우선으로 두는 것이 이 전략의 핵심 연결고리다.
- 표에 날짜가 없고 주차만 있는 문서만 학사일정 KB(현재 학기) 또는 needs_review로 간다.
- 이벤트 하이브리드의 날짜도 동일 우선순위를 따른다: 표의 실제 날짜 > (현재 학기)학사일정 KB > relative + needs_review.

**한 줄 요약: 제품은 현재·다가올 학기용이다. 교시표는 상위 학교로 넓게(싸다), 학사일정은 현재 학기만 최소로(비싸다), 문서 안에 이미 있는 날짜를 최우선으로 쓰고, 범위 밖은 needs_review로 남긴다. 두 사람의 시간을 과거 학사일정 롱테일에 태우지 마라.**
